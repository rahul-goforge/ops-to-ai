Week 1
LLM Foundations — How LLMs Actually Work
1.1 What is a Large Language Model (LLM)? ▼

Imagine you've read every book, every website, every Stack Overflow answer ever written. Now someone asks you a question, and you predict the most likely answer based on everything you've read. That's essentially what an LLM does — except with math instead of human memory.

An LLM is a massive neural network trained on enormous amounts of text. The training concept is simple: show the model a sentence with the last word removed, it tries to predict that word. Do this billions of times, adjusting the knobs each time it gets it wrong, and eventually the model gets really good at predicting what comes next.

🔑 Key Terms You MUST Know

Token
LLMs don't read words — they read "tokens." A token is a piece of a word. Kubernetes → [Kub][ern][etes] = 3 tokens. ~0.75 words per token. You PAY per token via APIs.
Context Window
The model's short-term memory. GPT-4o: 128K tokens. Claude: 200K tokens. This is why we need RAG — you can't dump an entire runbook into the prompt.
Temperature
Controls creativity (0–1). Temp 0 = always the most likely word. Temp 1 = creative. For DevOps: always use 0 or 0.1.
System Prompt
Instructions given BEFORE the user's question. Your most powerful tool. "You are a senior SRE. Always respond with exact kubectl commands."
Inference
The act of sending a prompt and getting a response back. You are NOT training the model — you're just using it. "Running inference" = calling the API.
Fine-tuning
Further training on your specific data. Expensive, complex, rarely needed. RAG achieves the same result 90% of the time. Ignore for now.
💡
The Transformer's secret weapon: self-attentionIn "The kubectl command failed because the service was not found in the production namespace", self-attention connects "failed" → "kubectl command". This is why LLMs can read a 50-line error log and figure out what went wrong. You don't need to understand the math.

1.3 Prompt Engineering — The 6 Techniques You Need ▼

Prompt engineering is the skill of writing instructions so the LLM gives you exactly what you want. This is the single most practical AI skill you'll use daily.

1. Role Prompting

prompt
System: You are a senior SRE with 10 years of experience managing
Kubernetes clusters on Azure AKS. You always provide exact commands.

2. Chain of Thought (CoT)

prompt
User: Analyze this error log and explain what went wrong.
Think step by step:
1. Identify the service
2. Identify the error type
3. Determine the root cause
4. Suggest the fix with exact commands

3. Output Format Control

prompt
User: List the top 5 Kubernetes best practices.
Return your answer as a JSON array with fields: "practice", "reason", "command"

4. Constraint Setting

prompt
System: Never guess. If you don't have enough information to answer,
say "I need more context" and list what you need.
Never suggest deprecated kubectl commands.
Always include the namespace flag in kubectl commands.

Week 2
RAG — Teaching AI Your Internal Knowledge
2.1 The Problem RAG Solves ▼

You ask an LLM "What's our rollback procedure for the payment service?" — and it makes something up, because it has never seen your internal documentation. LLMs are trained on public internet data. They know about Kubernetes in general, but nothing about YOUR infrastructure.

Option 1: Fine-tuning ❌
Retrain the model on your data. Expensive ($$$), slow, requires ML expertise. Overkill 99% of the time.
Option 2: Prompt Stuffing ❌
Paste your entire runbook into the prompt. Works until it exceeds the context window. Very expensive per query.
Option 3: RAG ✅
Search documents for relevant parts FIRST, then only send those parts to the LLM. Cheap, fast, scalable. This is what you'll build.
2.2 How RAG Works — The Full Pipeline ▼

RAG (Retrieval-Augmented Generation) is a 2-phase process:

Phase 1: Indexing (done once)

pipeline
Your Documents (runbooks, wiki, post-mortems)
        ↓
   [Document Loader]  — reads the files
        ↓
   [Text Splitter]    — breaks into small chunks (e.g., 500 chars each)
        ↓
   [Embedding Model]  — converts each chunk into a vector
        ↓
   [Vector Database]  — stores the vectors for fast similarity search

Phase 2: Querying (every user question)

pipeline
User Question: "How do I rollback a deployment?"
        ↓
   [Embedding Model]  — converts the question into a vector
        ↓
   [Vector Database]  — finds chunks whose vectors are most similar
        ↓
   [Top 4 chunks retrieved]
        ↓
   [Build Prompt] = System prompt + Retrieved chunks + User question
        ↓
   [LLM]             — generates answer using the retrieved context
        ↓
   Answer: "To rollback, run: kubectl rollout undo deployment/..."

2.3 Embeddings — How Computers Understand Meaning ▼

An embedding is a list of numbers (a "vector") that represents the MEANING of text. Similar meanings = similar numbers.

concept (simplified — real embeddings have 768–3072 dimensions)
"kubectl rollout undo"     → [0.82, 0.15, 0.91]
"rollback the deployment"  → [0.80, 0.17, 0.89]  ← Very similar!
"deploy to production"     → [0.31, 0.88, 0.22]  ← Very different
💡
Semantic similarity — even though "kubectl rollout undo" and "rollback the deployment" use different words, their vectors are close because they MEAN the same thing. This is the magic that makes RAG work.

2.5 Text Splitting (Chunking) — The Underrated Skill ▼

Chunking is how you break documents into pieces small enough for the vector database. Get this wrong and your RAG returns garbage.

chunking strategies
// 1. Fixed-size: split every 500 chars with 50-char overlap
Chunk 1: chars 0-500
Chunk 2: chars 450-950    ← 50-char overlap prevents cutting mid-sentence

// 2. Recursive character splitting (RECOMMENDED): try natural boundaries first
//    paragraph breaks → newlines → sentences → characters

// 3. Markdown/header-based: split on "## Heading" boundaries
//    Perfect for runbooks organized by section
⚠️
Best practices: Chunk size 500–1000 chars for runbooks. Always overlap (50–100 chars). Keep source metadata (filename, section) with each chunk. Test different sizes with your actual data.

Week 3
Agents, LangChain, and LangGraph
3.1 What is an AI Agent? ▼

An agent is an LLM that can TAKE ACTIONS, not just talk.

Regular LLM
"What's the CPU usage of the payment service?"

→ "I don't have access to your monitoring system. Here's how you could check: run kubectl top pods..."
AI Agent ✅
"What's the CPU usage of the payment service?"

→ calls prometheus_query tool → reads result → "The payment service is using 67% CPU across 3 pods..."

The 4 Components of an Agent

1. Brain (LLM)
The reasoning engine. Reads the question, decides what to do, interprets results.
2. Tools
Functions the LLM can call: check_pod_status(), query_prometheus(), search_runbooks()
3. Memory
Context from previous steps. The agent remembers what it already checked.
4. The Loop (ReAct)
Think → Act → Observe → Repeat. This cycle is the core of every agent.

The ReAct Pattern

THINK
Reason
→
ACT
Call Tool
→
OBSERVE
Read Result
→
THINK again
Loop or Answer

3.2 What is LangChain? (And LangChainGo?) ▼

LangChain is a framework providing pre-built building blocks for LLM applications. Think of it as Lego blocks — snap pieces together without writing all the plumbing from scratch.

Models
Unified interface to OpenAI, Anthropic, Ollama, Azure — same code, swap providers
Tools
Wrap any function so the LLM can call it
Retrievers
Connect to vector databases for RAG (pgvector, Qdrant, Weaviate, Chroma)
Text Splitters
Chunk documents for embedding
⬡
LangChainGo (github.com/tmc/langchaingo) is the Go port — 9,200+ GitHub stars. Pre-1.0 but stable for production DevOps tooling.

3.3 LangGraph — Graph-Based Agent Orchestration ▼

LangChain gives you chains (linear: A → B → C). But agents need to LOOP, BRANCH, and make DECISIONS. That's LangGraph — your agent modeled as a directed graph.

ReAct agent as a graph
[START]
   ↓
[Agent Node] ← LLM reads state, decides what to do
   ↓
{Should Continue?}
   ↓ YES (tool call)         ↓ NO (final answer)
[Tool Node]                 [END]
   ↓
   └──→ back to [Agent Node]
⚠️
Reality check for Go: LangGraph (Python) reached v1.0 in October 2025. There is NO official LangGraph in Go. Best alternative: ByteDance's Eino (github.com/cloudwego/eino) — 11K+ stars, battle-tested at TikTok.

For our project we write a clean custom agent loop in ~200 lines of Go. You learn more, and it works perfectly.

MCP, AIOps, and the Go AI Ecosystem
4.1 What is MCP (Model Context Protocol)? ▼

MCP is a new open standard (created by Anthropic, now maintained with Google) that defines how AI models connect to external tools and data sources.

🔌
The USB Analogy: Before USB, every device had its own proprietary connector. USB standardized it. MCP does the same for AI — it standardizes how an LLM connects to ANY tool.
architecture
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  MCP Client  │────→│  MCP Server  │────→│  External    │
│  (your agent)│←────│  (tool host) │←────│  Service     │
└──────────────┘     └──────────────┘     └──────────────┘
                          ↑
                     Exposes tools like:
                     - list_pods()
                     - query_prometheus()
                     - create_jira_ticket()
⬡
MCP in Go is excellent. The official Go SDK (modelcontextprotocol/go-sdk v1.0+) is one of the best MCP implementations in any language. A DevOps MCP server compiles to a single binary — perfect for containers and sidecars.

Phase 5Build the Agent Loop — internal/agent.go▼

The heart of InfraBot — the ReAct agent loop. User question → RAG search → LLM → tool call → observe → loop → final answer.

go — ReAct loop (internal/agent.go)
// Step 1: Search the knowledge base (RAG)
docs, _ := a.rag.Search(ctx, userQuery, 4)

// Step 2: Build messages with system prompt + RAG context
messages := buildMessages(systemPrompt, ragContext, userQuery)

// Step 3: The ReAct Loop (max 5 iterations)
for i := 0; i < 5; i++ {
    response := a.llm.GenerateContent(ctx, messages)
    toolCall, isTool := parseToolCall(response)
    if !isTool { return response } // Final answer
    
    // Execute the tool and feed result back to LLM
    toolResult := executeTool(toolCall)
    messages = append(messages, toolResult)
}
